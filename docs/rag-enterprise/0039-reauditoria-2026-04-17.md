## 4. Reranking Opcional (BM25F)

### Como Funciona

O reranker BM25F é aplicado após a fusão RRF e os filtros, mas antes do corte `top_k`. Ele reordena os candidatos usando:

1. **TF-IDF (BM25)**: para cada termo da query, calcula a frequência do termo no chunk com normalização por tamanho do documento
2. **IDF por corpus**: computado dinamicamente a partir dos candidatos retornados pela busca (não usa corpus externo)
3. **Decaimento posicional**: termos que aparecem no início do texto têm peso maior
4. **Campos ponderados**: texto do chunk (peso 1.0), nome do arquivo (peso 0.3), tags (peso 0.2)
5. **Blend com confiança original**: `score_final = sqrt(confidence_original * bm25f_score)` — equilibra relevância sem descartar sinais da busca vetorial

### Como Habilitar

```python
# Via API (requer reranking=True na requisição)
SearchRequest(query="...", reranking=True)

# Via variável de ambiente (global, para toda a aplicação)
RERANKING_ENABLED=true  # em .env ou export
```

### O que é Exposto na Resposta

| Campo | Descrição |
|---|---|
| `reranking_applied` | `True` se reranking foi executado |
| `reranking_method` | `"bm25f"` ou `"neural"` quando aplicado |
| `result.source` | `"dense+sparse_rrf"` (sem reranking), `"dense+sparse_rrf+bm25f"` (BM25F), ou `"dense+sparse_rrf+neural"` (neural) |

### Testes

```
8 testes cobrindo: compatibilidade sem reranking, reordenação, metadata, standalone BM25F, edge cases
```

### Limitações

- **Reranking neural não implementado** — cross-encoder ou re-ranker neural (Cohere, etc.) não existe. BM25F é lexical e não captura similaridade semântica.
- **IDF calculado sobre candidatos** — não sobre todo o corpus. Pode ser menos preciso para queries com poucos matches.
- **Sem impacto na avaliação HIT@K** — BM25F reordena resultados para relevância percebida, mas HIT@K mede recall de documento, não a ordem. Não deve alterar HIT@K significativamente.

### Interpretação dos resultados

**Para este dataset (30 perguntas sobre políticas_fluxpay.md, HIT@5=100% baseline):**

- **HIT@K inalterado**: O dataset é bem respondido pelo retrieval base. BM25F não melhora o recall de documento porque o corpus pequeno faz o embedding já retornar o documento correto no topo.
- **Latência aumenta**: BM25F adiciona ~130ms porque re-scora todos os candidatos (não apenas top-k) antes do corte.
- **low_confidence_rate aumenta com reranking**: O blend com BM25F pode reduzir scores dechunks que têm alta confiança vetorial mas baixa sobreposição léxica com a query, haciendo que algunos ultrapassem o `quality_floor` threshold (0.70 por padrão). Este efeito é observável em ~57% das queries neste dataset.
- **avg_score cai**: O `confidence_score` normalizado (0-1) é substituído pelo blend no variant, que pode ser menor para chunks sem sobreposição léxica forte.

**Para datasets maiores ou mais diversificados**, o BM25F tende a reorder resultados na direção correta sem afetarHIT@K, mas com latência adicional.

### Limitações conhecidas

1. **BM25F lexical**: não captura similaridade semântica — queries com termos diferentes das políticas mas sentido equivalente não se beneficiam.
2. **IDF calculado sobre candidatos**: não sobre todo o corpus. Pode ser menos preciso para queries com poucos matches.
3. **Sem impacto em HIT@K neste dataset**: o corpus pequeño faz o embedding retrieval já ser perfeito; BM25F só reordena, não recupera documentos adicionais.
4. **Latência adicionada**: o reranking processa todos os candidatos antes do corte top-k, adicionando ~5-10ms por кандидат.
5. **quality_floor** pode marcar mais queries como baixa confiança quando reranking reduz scores — isso é esperado e desejado (indica que BM25F considera o resultado menos relevante lexicalmente).

## 8. Semantic Chunking Opcional

### O que foi implementado

**`semantic_chunk()`** em `chunker.py` — chunker que agrupa sentenças com base em similaridade semântica:

```
semantic_chunk(normalized_doc):
  1. Split em sentenças (regex [.!?]\s+)
  2. Greedy grouping por tamanho (chunk_size=300 chars, menor que recursive)
  3. Embed grupos → compute cosine similarity entre grupos consecutivos
  4. Break antes de grupo quando similarity < avg * coherence_threshold (0.7)
  5. Fallback offline: retorna greedy groups sem coherence breaks
```

**Parâmetros:**
- `chunk_size=300` (vs recursive=1000) — sentenças são menores
- `coherence_threshold=0.7` — fração da média de similaridade abaixo da qual quebra

### Como habilitar

```python
# Via API multipart form
POST /documents/upload
  file: <file>
  chunking_strategy: semantic  # vs "recursive" (default)

# Via reindex
reindex_document(document_id, workspace_id, chunking_strategy="semantic")

# No corpus existente (re-chunk)
```

### Diferenças vs chunking recursive

| Aspecto | `recursive` | `semantic` |
|---|---|---|
| Granularidade | párgrafos/sentenças (1000 chars) | sentenças (300 chars) |
| Coherence break | por tamanho | por similaridade semântica |
| Embbeddings | não | sim (offline fallback) |
| Chunk count | menor | maior |
| overlap | 200 chars | 50 chars |

### Validação estrita de strategy

`ingest_document()` e `upload_document()` validam `chunking_strategy` contra `VALID_CHUNKING_STRATEGIES = {"recursive", "semantic"}`. Valores inválidos retornam erro 400 com mensagem descritiva.

### Metadata exposta

- `DocumentUploadResponse.chunking_strategy` ✅
- `DocumentMetadata.chunking_strategy` ✅ (via chunks[0].strategy)
- `Chunk.strategy` ✅ (`"semantic"` ou `"recursive"`)

### Comparação A/B de Chunking

**Endpoint:** `POST /evaluation/chunking-ab`

**Parâmetros:**
- `document_id` (obrigatório): documento a avaliar
- `workspace_id` (default: `default`)

**Fluxo operacional:**
1. Detecta estratégia ORIGINAL e captura chunks originais do disco (antes de qualquer modificação)
2. Backup dos chunks correntes em `*_chunks_backup.json`
3. Avaliação baseline com `recursive`
4. Reindex com `semantic`
5. Avaliação variant com `semantic`
6. Restauração FIEL: copia `*_chunks_backup.json` → `*_chunks.json` e re-indexa Qdrant com os chunks originais (não re-chunk a partir do raw)
7. Verifica que o arquivo em disco é idêntico ao backup → `corpus_restored`


**`embedding_status`** (campo novo em `ChunkingABResponse`):
- `"original"` — backup já tinha embeddings válidos; foram reutilizados como estavam
- `"regenerated"` — backup não tinha embeddings; foram gerados de `chunk.text` via `get_embeddings_batch`
- `"degraded_zero_fallback"` — backup não tinha embeddings e `get_embeddings_batch` também falhou (offline); Qdrant foi indexado com vetores `[0.0] * 1536` — retrieval degradado

**Garantia de integridade (restauração fiel via backup):**
- `corpus_restored: true` é retornado quando `chunk_id`, `text`, `strategy` e `chunk_index` do arquivo em disco são idênticos aos do backup — os campos que importam para retrieval estão fiéis ao original
- `embedding_status` indica a qualidade dos vetores restaurados no Qdrant — não é mais invisível
- O campo `embedding` pode estar presente no arquivo restaurado com vetores válidos (regenerados a partir do texto dos chunks) — isso é correto: Qdrant é restaurado com embeddings úteis, não zeros
- Se o backup já continha embeddings válidos, eles são reutilizados → `embedding_status="original"`
- Se qualquer chunk tem embedding ausente ou inválido, **todos** os chunks são regenerados via `get_embeddings_batch` → `embedding_status="regenerated"`
- **Proteção contra truncamento**: se `get_embeddings_batch` retornar uma quantidade diferente de chunks, `RuntimeError` é levantada com mensagem descritiva — sem truncamento silencioso
- **Validação por chunk**: `_restore_embeddings_if_needed` valida cada embedding individualmente (tipo, comprimento, valores numéricos); se qualquer regeneração produzir embedding inválido, `RuntimeError` é levantada
- Se `get_embeddings_batch` falhar (offline), os chunks são retornados sem modificação e `reindex_document` indexing com zeros — retrieval degradado; operador pode reindexar manualmente depois
- `corpus_restored: false` se a restauração falhar ou se os campos semânticos divergirem do backup
- Caminho de erro: tenta restauração fiel via `reindex_document(..., chunks=original_chunks)`, que re-indexa Qdrant com os embeddings originais ou regenerados
- Documentos originalmente `semantic` voltam para `semantic` — a estratégia é detectada a partir do arquivo original e o backup guarda os chunks exatos daquela estratégia
- Backup é removido ao final (sucesso ou erro)

** Métricas disponíveis:**
- `chunk_count`: número de chunks por estratégia (indica granularidade)
- `HIT@1`, `HIT@3`, `HIT@5`: qualidade de retrieval
- `avg_latency_ms`: latência média de retrieval
- `avg_score`: score médio do top-1
- `low_confidence_rate`: fração de queries com baixa confiança
- `winner`: `"baseline"` | `"variant"` | `"tie"` baseado em HIT@5

### Limitações

1. **offline fallback de embeddings na restauração (último recurso, agora sinalizado)** — `embedding_status="degraded_zero_fallback"` só ocorre quando tanto Qdrant quanto a API de embeddings falham. O sistema tenta: (a) ler embeddings diretamente de Qdrant por chunk_id; (b) regenerar via `get_embeddings_batch`; (c) só então usar zeros. O operador pode reindexar manualmente depois.
2. **offline fallback não quebra coherence breaks** — quando embeddings estão indisponíveis (offline), o semantic chunker retorna chunks greedy sem coherence breaks, comportamento equivalente a grouping sentence-level sem medida de similaridade.
3. **reindex_document() persiste chunks** — após reindexar, chunks são salvos em `*_chunks.json` e o `document_registry` reflete a estratégia corrente.
4. **Comparação usa o dataset da workspace** — se não houver `dataset.json` na workspace, o endpoint retorna 404.
5. **chunk_count varia com a estratégia** — semantic tipicamente produz mais chunks (sentenças curtas) que recursive, isso é esperado e não indica erro.
6. **Backups de A/B incluem embeddings de Qdrant** — antes de iniciar a avaliação A/B, o sistema lê embeddings de Qdrant para os chunks atuais e os injeta no arquivo de backup. Isso garante que a restauração consiga recuperar os vetores originais mesmo que a API de embeddings esteja offline no momento da restauração.

## 7. Reranking Neural (Cross-Encoder)

### O que foi implementado

**`NeuralReranker`** em `vector_service.py` — reranker que usa embeddings para calcular similaridade coseno entre query e candidatos:

```
rerank(query, candidates):
  1. Embed query → query_emb
  2. Embed cada candidato → cand_emb_i
  3. sim_i = cosine(query_emb, cand_emb_i)
  4. blend_i = sqrt(confidence_score * sim_i)
  5. Ordena por blend_i desc, tie-break por original_score
```

**Fallback seguro**: se `get_embedding` falhar (offline, sem API key), retorna candidatos inalterados — não quebra o pipeline.

**Cache in-process**: embeddings iguais são reutilizados dentro da mesma requisição.

### Interface pluggable

`_get_reranker(method)` retorna o reranker correto por método:

| method | classe |依赖 |
|---|---|---|
| `"bm25f"` | `BM25FReranker` | nenhuma (local) |
| `"neural"` | `NeuralReranker` | `embedding_service` (offline fallback existe) |
| `"none"` | `None` | — |

### Seleção de método

- **Request-level**: `SearchRequest.reranking_method="neural"` ou `QueryRequest.reranking_method="neural"`
- **Global config**: `RERANKING_METHOD=neural` em `.env`
- **Validação na API**: `/evaluation/ab?variant_method=neural` aceita apenas `"bm25f"` ou `"neural"`

### Como habilitar

```bash
# Via variável de ambiente (global)
RERANKING_METHOD=neural  # em .env

# Via API por requisição
SearchRequest(query="...", reranking=True, reranking_method="neural")

# A/B comparison
POST /evaluation/ab?variant_method=neural
```

### Comparação de métodos disponíveis

| Aspecto | `none` | `bm25f` | `neural` |
|---|---|---|---|
| Tipo | sem reranking | lexical (TF-IDF) | embedding-based |
| Dependência externa | nenhuma | nenhuma | `embedding_service` |
| Fallback offline | — | — | retorna inalterado |
| Latência adicional | 0ms | ~130ms | ~5ms (offline) ou mais |
| Similaridade semântica | ✅ | ❌ | ✅ |
| Sensível a termos não no corpus | ❌ | ❌ | ✅ |

Campo `reranking_method` agora expõe `"bm25f"` ou `"neural"` (antes só `"bm25f"`).

### Recuperação de embeddings na restauração

A restauração de embeddings segue esta ordem de prioridade:

1. **Chunks já têm embeddings válidos?** → usa direto, sem qualquer operação (`original`)
2. **Qdrant (offline-safe)**: tenta ler diretamente de Qdrant usando `chunk_id`. **Cada embedding é validado** (tipo `list`, dimensão `EMBEDDING_DIM`, valores numéricos) antes de ser aceito. Embeddings inválidos são tratados como "não encontrados" — não mascaram o fallback para API.
3. **API de embeddings**: para chunks não encontrados ou com embeddings inválidos em Qdrant, chama `get_embeddings_batch` para regenerar a partir do texto.
4. **Zeros (último recurso)**: se tanto Qdrant (leitura ou validação) quanto a API falharem, `embedding_status="degraded_zero_fallback"` é retornado e zeros são indexados em Qdrant.

**Garantia de backup**: antes de iniciar uma avaliação A/B, o sistema lê embeddings de Qdrant para os chunks atuais e os injeta no arquivo de backup. Isso significa que o backup é auto-contido: a restauração consegue recuperar os vetores mesmo que a API de embeddings esteja offline no momento da restauração (desde que Qdrant esteja disponível).

### Interpretação

- **`total_reindex_operations`** (eventos) vs **`current_corpus.by_strategy`** (estado atual): um A/B de um único documento gera 3 eventos de reindex, mas o corpus continua com 1 documento. Use `unique_documents_reindexed` para saber quantos documentos distintos foram afetados.
- **`current_corpus.by_strategy`**: reflete o estado real do corpus em disco no momento da requisição — não é afetado por eventos passados.
- **`total_degraded_zero_fallback > 0`**: indica problemas sérios — tanto Qdrant quanto a API de embeddings falharam. Operador deve investigar urgentemente.
- **`embedding_status="original"`**: os vetores vieram de Qdrant (leitura direta, sem regeneração) ou já estavam no backup. Ambos são estados de alta qualidade.
- **`embedding_status="regenerated"`**: vetores gerados via API a partir do texto. Qualidade depende da API.
- **`historical.by_source`**: permite distinguir reindex manuais de operações automáticas de A/B.
- **`avg_processing_time_ms` por estratégia**: semantic é tipicamente mais lento que recursive.

### O que foi implementado

#### Campos de telemetria por query (3 estados)

Cada query agora loga em `queries.jsonl` com três campos que cobrem todos os estados:

| Campo | Descrição |
|---|---|
| `query_expansion_requested` | `true` quando expansion foi habilitada (por param ou config) |
| `query_expansion_applied` | `true` quando expansion foi solicitada E produziu texto útil |
| `query_expansion_fallback` | `true` quando expansion foi solicitada MAS retornou vazio/truncada |

Os três estados são mutuamente exclusivos:

- **requested_but_fallback**: `query_expansion_requested=True AND query_expansion_fallback=True`

#### Como interpretar

**Custo da feature:**
- `avg_expansion_latency_ms` (em `requested_and_applied`) — quanto do tempo total é gasto no passo HyDE
- Overhead = `requested_and_applied.avg_latency_ms - not_requested.avg_latency_ms`

**Benefício (usar A/B para validação):**
- Comparar `avg_score_top1` e `low_confidence_rate` entre `requested_and_applied` e `not_requested`
- **Cuidado**: esse grupo não é controlado — o viés de seleção é alto em produção. `/evaluation/query-expansion-ab` é a forma correta de medir benefício.

**Fallback (`requested_but_fallback`):**
- `fallback_count` = queries onde expansão foi habilitada mas retornou vazio/truncada
- `fallback_rate = fallback_count / total_expansion_requests` — **denominador = expansion requests**, não total de queries
- Causas: API key ausente, timeout, ou LLM retornou resposta trivial (< 10 chars)
- `requested_but_fallback.avg_latency_ms ≈ not_requested.avg_latency_ms` — quando expansion falha, o overhead é ~zero (retorna rápido)

**Quando expansion está ajudando ou só adicionando custo:**

| Sinal | Interpretação |
|---|---|
| `requested_and_applied.avg_score_top1` > `not_requested.avg_score_top1` | Expansão pode estar ajudando retrieval |
| `requested_and_applied.low_confidence_rate` < `not_requested.low_confidence_rate` | Expansão pode estar reduzindo dúvidas |
| `requested_and_applied.avg_expansion_latency_ms` > 500ms | Custo alto; considerar desligar se HIT@K não melhorar |
| `fallback_rate` > 5% | Expansion falhando frequentemente; verificar API key |
| `requested_but_fallback` presente com `fallback_count > 0` | Fallback detection funcionando corretamente |

## 12. Query Expansion — Política Adaptativa

### O que foi implementado

Adicionada política adaptativa para query expansion com três modos de operação:

| Modo | Comportamento |
|---|---|
| `"off"` | Expansão nunca é solicitada |
| `"always"` | Expansão é sempre solicitada quando habilitada globalmente |
| `"adaptive"` | Heurística determinística decide por query (default quando `QUERY_EXPANSION_ENABLED=true`) |

O modo é controlado pelo campo `query_expansion_mode: "off" | "always" | "adaptive"` em `QueryRequest` e `SearchRequest`. Para compatibilidade, `query_expansion: true/false` (legacy) continua funcionando mapeando para `always/off`.

### Heurística adaptativa (modo `"adaptive"`)

Regras avaliadas em ordem:

1. **Skip — specific_lookup**: expansão é pulada para queries que parecem identificadores específicos:
   - Números com ≥4 dígitos (anos, IDs, telefones)
   - Padrões tipo data/ratio (`\d+/\d+`)
   - Códigos com prefixo maiúsculo + dígito (ex: `PROJ123`)
   - Termos de referência: `art.`, `artigo`, `nº`, `número`, `protocolo`, `ref.`, `processo`

2. **Expand — short_query**: expansão é aplicada para queries curtas:
   - Menos de 3 palavras, OU
   - Menos de 12 caracteres

3. **Expand — natural_language_question**: expansão é aplicada para perguntas em linguagem natural:
   - Contém `?`, OU
   - Começa ou contém palavra interrogativa: `o que`, `qual`, `quem`, `quando`, `onde`, `como`, `por quê`, `what`, `who`, `when`, `where`, `how`, `why`, `which`, `does`, `is there`, `are there`, `can i`, `posso`, `pode`, `existe`

4. **Default — general_query**: expansão é aplicada para queries abertas genéricas

### Campos de telemetria para política adaptativa

Dois novos campos em `QueryLogItem`, `SearchResponse`, e `QueryResponse`:

| Campo | Descrição |
|---|---|
| `query_expansion_mode` | Modo ativo: `"off"`, `"always"`, ou `"adaptive"` |
| `query_expansion_decision_reason` | Razão da decisão: `always_mode`, `adaptive:specific_lookup`, `adaptive:short_query`, `adaptive:natural_language_question`, `adaptive:general_query` |

### Compatibilidade com lógica anterior

```python
# Request com query_expansion_mode explícito (novo)
QueryRequest(query="...", query_expansion_mode="adaptive")

# Request com query_expansion legacy (mantido)
QueryRequest(query="...", query_expansion=True)  # → mapeia para "always"
QueryRequest(query="...", query_expansion=False)  # → mapeia para "off"

# Request sem indicação (default)
QueryRequest(query="...")  # → usa QUERY_EXPANSION_ENABLED global
# Se QUERY_EXPANSION_ENABLED=true → "adaptive", senão → "off"
```

### Exemplos de decisão

| Query | Modo | Decisão | Razão |
|---|---|---|---|
| `protocolo 12345` | adaptive | skip | `specific_lookup` |
| `artigo 14 da lei` | adaptive | skip | `specific_lookup` |
| `PROJ2024001 status` | adaptive | skip | `specific_lookup` |
| `2024-01-15 prazo` | adaptive | skip | `specific_lookup` |
| `Qual o prazo?` | adaptive | expand | `natural_language_question` |
| `O que é Fluxpay?` | adaptive | expand | `natural_language_question` |
| `prazo?` | adaptive | expand | `short_query` |
| `Fluxpay` | adaptive | expand | `short_query` |
| `Explique onboarding` | adaptive | expand | `general_query` |
| `protocolo 12345` | always | expand | `always_mode` |
| `Qual o prazo?` | off | skip | — (nunca solicitado) |
